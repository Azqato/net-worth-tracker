# Net Worth Tracker — Security

## Authentication Model

There is no authentication. The app does not identify users, create sessions, or issue tokens of any kind.

The `authModal` in `app.html` renders a full sign-up UI (Google, Apple, X OAuth buttons + email/password form) but all handlers are stubs that do nothing. This UI exists as a design placeholder and is not wired to any backend. It should be considered dead code until a real auth backend is implemented.

---

## Authorization Model

There are no user roles or permissions. Anyone who opens the app in a browser has full read/write access to the data stored in that browser's `localStorage` for this origin. Data is scoped to the browser profile and origin — a different browser, incognito window, or device has no access to another session's data.

---

## Data Storage

| Data | Where stored | Protection |
|---|---|---|
| Snapshot data (`networth_snapshots`) | Browser `localStorage` | Origin-scoped by browser; inaccessible to other origins via same-origin policy |
| Inactivity dismiss count (`inactivity_dismiss_count`) | Browser `localStorage` | Same as above |
| No other user data | — | — |

**No data is transmitted to any server.** The app makes zero outbound network requests with user data. The only outbound requests are CDN fetches for Chart.js and SheetJS scripts — these are GET requests for static JavaScript files with no user data in the request.

**localStorage is not encrypted.** Any script running on the same origin could read the data. Since this is a GitHub Pages static site, there is no server-side code, and all JavaScript is visible in the browser. Users with malware or malicious browser extensions on their device should be aware that `localStorage` values are readable.

**No server-side storage.** There is no database, no file system, and no cloud storage. The developer cannot access user data.

---

## Environment Variables

There are no environment variables. This is a fully static site with no server-side configuration.

No secrets, API keys, or credentials are used anywhere in the codebase. Nothing should ever be committed that resembles a secret — there is no reason for one to exist in this project.

## Third-Party Trust

| Service | What it receives | What it does |
|---|---|---|
| jsDelivr CDN (`cdn.jsdelivr.net`) | Browser IP, User-Agent (standard HTTP headers) | Serves `chart.js` static JavaScript file |
| SheetJS CDN (`cdn.sheetjs.com`) | Browser IP, User-Agent | Serves `xlsx.full.min.js` static JavaScript file |
| GitHub Pages (`azqato.github.io`) | Browser IP, User-Agent | Serves all HTML, CSS, and JS files |
| `azqato.github.io/support.html` | Browser IP, User-Agent (only if Support link is clicked) | External support page; separate site |

**No financial data is sent to any of these services.** CDN requests are purely for fetching JavaScript. The user's snapshot data never leaves the device.

**Subresource Integrity (SRI)**: The CDN `<script>` tags in `app.html` do not currently use `integrity` attributes. Without SRI hashes, a compromised CDN could theoretically serve malicious JavaScript. This is a known risk for CDN-served dependencies. Mitigation: use well-established CDNs with strong track records (jsDelivr, SheetJS official CDN) and monitor for any CDN security incidents.

---

## Known Attack Surface

| Area | Risk | Mitigation |
|---|---|---|
| CDN dependency injection | A compromised CDN could serve malicious Chart.js or SheetJS, which would run in the user's browser | SRI hashes not implemented (see technical debt in TRD); use trusted CDNs; monitor for incidents |
| Excel import parsing | SheetJS parses user-provided `.xlsx` files; malformed files could trigger parsing bugs | No code execution from imported files; SheetJS is sandboxed to JavaScript; risk is low but present |
| localStorage access by browser extensions | Malicious browser extensions can read `localStorage` | No mitigation possible at the app level; user responsibility |
| Inactivity modal `authModal` stub | `authModal` collects email/password into form inputs, but `createAccount()` and `oauthSignIn()` are stubs that do nothing — no data is stored or transmitted | Current behavior is safe; risk increases if the stubs are ever wired to a real backend without a security review |
| XSS via snapshot history rendering | `renderSnapshotHistory()` writes directly to `innerHTML` using data from `localStorage`; if `localStorage` is tampered, arbitrary HTML could be injected | Data written to `localStorage` is always generated by this app's own functions (number formatting, date strings); no user-supplied raw HTML is stored. Risk is low but not zero if another script writes to the `networth_snapshots` key. |

## Dependency Policy

| Practice | Status |
|---|---|
| CDN dependency versions are pinned | Yes — Chart.js `@4.4.0` and SheetJS `@0.20.1` are pinned in `<script src>` URLs |
| Dependencies are monitored for vulnerabilities | No automated monitoring (no npm, no Dependabot); manual check required |
| Dependency updates | Manual — update CDN URL version when a security patch is released for a pinned dependency |
| SRI hash validation | Not implemented; recommended for future hardening |

There are no npm dependencies, no `package.json`, and no `node_modules`. The dependency surface is limited to two CDN-served JavaScript libraries. Monitor the Chart.js and SheetJS GitHub repositories for security advisories and update the pinned CDN URLs when a patch is released.
